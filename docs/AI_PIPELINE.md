# The APEX AI Pipeline (Reasoning & Context Engineering)

This document breaks down the multi-stage AI reasoning pipeline that powers APEX's Decision Intelligence.

## 1. The Core Philosophy: "Constrained Generation"
LLMs hallucinate when asked to invent solutions from scratch. APEX does not ask the LLM to invent a solution. Instead, APEX asks the LLM to *evaluate* mathematically possible paths generated by the deterministic Causal Graph.

## 2. Pipeline Stages

### Stage 1: Telemetry & Trigger
* **Input:** Real-time sensor anomaly (e.g., `Sensor ID: S-301`, `Metric: Pressure`, `Status: High`).
* **Process:** The Event Listener receives this and immediately queries the NetworkX Graph DB.

### Stage 2: Graph Traversal & Blast Radius (Deterministic)
* **Process:** NetworkX runs a Breadth-First Search (BFS) starting at `S-301`. It identifies all downstream equipment (e.g., `Heat Exchanger E-102`) and upstream dependencies.
* **Output:** A JSON subgraph representing the "Blast Radius".

### Stage 3: Context Retrieval (RAG)
* **Process:** For every equipment ID in the Blast Radius, APEX queries the Qdrant Vector Database.
* **Embeddings:** We use `text-embedding-3-small` to retrieve:
  1. Operating Limits (from OEM Manuals).
  2. SOPs for isolation.
  3. Historical incident resolutions for these specific assets.

### Stage 4: Synthesis & Path Evaluation (LLM Reasoning)
* **Process:** The Blast Radius JSON and RAG Context are injected into the LLM Prompt.
* **Prompt Engineering Strategy:** We use a *Chain-of-Thought (CoT)* prompt with a *Persona*.
  * *Persona:* "You are a Principal Reliability Engineer at a Level-4 hazardous chemical plant."
  * *Task:* "Evaluate the following 3 graph-derived mitigation paths based on the provided safety regulations. Output your decision as a structured JSON object detailing the optimal path, a confidence score, and a brief rationale."
* **Output:** The LLM returns a structured JSON payload containing the chosen path.

### Stage 5: Runbook Compilation
* **Process:** A deterministic function takes the LLM's chosen path and maps it to specific UI components. For instance, if the LLM chose "Isolate Valve V-10", the function generates a `ChecklistStep` component with a `requires_loto: true` flag.

## 3. Dynamic Adaptation (The Feedback Loop)
If an operator marks a runbook step as `Failed` (e.g., Valve stuck), the UI sends a `POST` request back to Stage 2.
* The Graph is updated: `V-10.status = "unavailable"`.
* The Pipeline runs again from Stage 2. Since `V-10` is unavailable, NetworkX finds an alternative path (e.g., upstream at `V-12`). The Runbook regenerates in < 3 seconds.
